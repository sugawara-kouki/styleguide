TypeScript Style Guide

命名規則

変数

  • キャメルケースで記述すること
  • 型がboolの場合、is,can,hasのいずれかを接頭辞につけること
  • 名前の短さではなく分かりやすさを優先して命名すること
  • 配列やオブジェクトなど、複数のデータがある場合は名前も複数形とすること
  • for,whileでのインデックスとして使う変数はi, j, kを使用すること
    ※インデックスの使用頻度が高い場合は1文字ではなくわかりやすい名前を付けること
  • 関数からの返却値を受け取る場合、関数名に合わせた変数名にすること
    var currentPageName = getCurrentPageName();
  • flg,cntなど、省略形での命名は基本的にしないこと
    flg -> flag, cnt -> count

定数

  • すべて大文字のスネークケースで記述すること
  • HOGEHOGE_1, HOGEHOGE_2のように一目見てデータが判断できない命名はしないこと

関数

  • キャメルケースで記述すること
  • 名前の短さではなく分かりやすさを優先して命名すること
  • 実行される処理内容と一致する命名をすること
  • 値をreturnする関数の場合は、返却値が明確な命名をすること

クラス

  • パスカルケースで記述すること
  • 名前の短さではなく分かりやすさを優先して命名すること

インターフェース

  • パスカルケースで記述すること
  • メンバはキャメルケースで記述すること
  • プレフィックスにIをつけないこと

タイプ

  • パスカルケースで記述すること
  • メンバはキャメルケースで記述すること

名前空間 (namespace)

  • パスカルケースで記述すること

Enum

  • パスカルケースで記述すること
  • メンバもパスカルケースで記述すること

ファイル

  • キャメルケースで命名すること
  • 名前で何が記載されているのかがわかるように命名すること

記述規則

コメント

  • コードの内容をなぞるだけのコメントは記載しないこと
  • 命名が不適切でコメントを記述する必要がある場合はリファクタリングを行うこと

繰り返し処理

  • 配列操作についてはmap, filterなどの配列用関数を使用すること
  • インデックス値を使用しない場合は、拡張for文を使用すること
    for-of, for-in
  • forの中でifを記載する場合は早期continueでネストが深くなるのをを回避すること

条件分岐処理

  • 肯定条件を先に記述すること
  • ガード節を用いてelseはできるだけ記述しないこと
  • 厳密等価演算子を使用すること
  • if (!hogehoge)は可能な限り使用しないこと
    if (hogehoge === false)
  • 複雑な条件式を使用している場合は逆の発送を使用して簡潔ならないかを検討すること

関数

  • 関数には1つまたは最小単位の機能のみ実装すること
  • 戻り値, 引数の型定義を行うこと
  • フラグ引数はできるだけ使用しないこと 
    const action = (foo:boolean) => {
  if(foo){
    // Do ActionA
  } else {
    // Do ActionB
  }
}

const hoge = () => {
  var foo:boolean;
  // Process
  Action(foo);
}
    const actionA = () => {
  // Do ActionA
}

const actionB = () => {
  // Do ActionB
}

const hoge = () => {
  var foo:boolean;
  // Process
  if(foo){
    actionA();
  } else {
    actionB();
  }
}

ネスト

  • ネストは最大3つまでに収めること
    どうしても3つ以上になってしまう場合はリファクタリングを検討すること
  • 複数のif文でネストされるときは条件を見直し、早期リターンできないか検討すること

配列

  • 配列のアノテーションはfoos: Array(hoge)ではなくfoos: Foo[]として付けること
  • オブジェクトが配列かどうか検証する際は、Array.isArray(arr);を使用すること

型定義

  • 基本的にはtypeを使用すること
  • extendやimplementsを使用する場合はinterfaceを使用すること

三項演算子

  • 基本的に使用しないこと
  • 使用する場合は簡単な条件式であることを確認すること

修飾子

  • varは基本的に使用しないこと
  • letは値が変更される可能性があるため、基本的にはconstを使用すること

フォルダ構成

構成規則

  • 基本的に使用するフレームワークで推奨されている構成を使用すること
  • utils, constants, types等はPJ共通で構成すること
